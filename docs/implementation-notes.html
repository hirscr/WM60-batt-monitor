WM_controller — Implementation Notes

Decisions, tradeoffs, and non-obvious choices made during build. Append new entries at the top.

2026-05-25 — PV prediction log: upsert + self-healing tick + backfill --from-state

Files: services/pv_prediction_logger.py, tools/backfill_pv_actual.py, tests/test_pv_prediction_log_csv.py.

What changed

  • PVPredictionLogger._append_row is now an upsert keyed by date. It reads the entire CSV into memory, replaces the row whose date field matches (preserving its position), or appends a new tail row, and rewrites the file via a tempfile + os.replace. The old append-only path could not heal a row that was written prematurely with blank columns.
  • New helpers: _build_row (pure formatter), _read_all_rows, _atomic_rewrite (tempfile.mkstemp into the same directory, fsync, then atomic rename), and _is_today_row_complete (predicate the tick consults).
  • _tick no longer early-returns purely on last_pv_log_date >= today_local. It now treats that flag as a hint and re-runs after sunset+30min whenever today's on-disk row is incomplete. The no-op condition is "row complete AND flag == today". last_pv_log_date is set only after a successful, fully-populated upsert; an incomplete row leaves the flag at its previous value so the next tick re-attempts.
  • _tail_date is removed — the upsert replaces its only use site.
  • tools/backfill_pv_actual.py ported to argparse and gained --from-state. Default behavior (no flags) is unchanged: recompute actual_kwh from the battery log. --from-state alone patches the four gate-context columns from wm_state.json and leaves actual_kwh untouched. --from-state --actual does both in one invocation.
  • The backfill tool refuses (exit code 3) if weather_gate_evaluated_date in wm_state.json does not match the target date — never writes stale gate context.
  • Seven new tests in tests/test_pv_prediction_log_csv.py cover the upsert path (replacement-in-place, row-position preservation), the atomic-write pattern (no leftover temp file, rollback on failure), and the tick policy (self-heals incomplete rows, no-ops when complete, holds the flag when row stays incomplete). Test count: 173 (up from 166).

Non-obvious decisions

  • Method name _append_row retained despite semantics being upsert. The existing test surface calls logger._append_row(...) as the cheapest way to exercise the write path in isolation. Renaming would have churned six existing tests for no functional gain. The docstring is explicit that the semantic is upsert; "_append_row" is now a historical alias.
  • actual_kwh == 0 counts as incomplete, even though 0.0 is a real cloudy-day reading. The spec is explicit on this and the May-25 motivation cannot be satisfied any other way: the broken row already on the Pi has actual_kwh=0.00 and must self-heal at tonight's sunset+30min. The cost is that on a legitimately cloudy day, the tick re-writes the same 0.0 row every tick interval until midnight. Wasteful but bounded; the on-disk row stays at 0.0000 the whole time. Captured in test_tick_does_not_advance_flag_when_row_stays_incomplete as a deliberate, documented behavior.
  • Per-column gate-context completeness, conditional on wm_state. A blank gate-context column on disk does NOT count as incomplete unless wm_state.json has the corresponding weather_gate_* value set AND weather_gate_evaluated_date == today. This means a row written on a day when the gate didn't evaluate (e.g. weather gate disabled) is considered complete with blank gate context. Otherwise the tick would never settle on those days.
  • Completeness is checked twice in _tick. Once before any work (the no-op short circuit) and once after the upsert (to decide whether to advance last_pv_log_date). Both reads are cheap (a single CSV slurp). The second check uses the wm_state snapshot from the top of the tick, intentionally — if the gate commits new values mid-tick, those land next time around.
  • Atomic-write target dir is the prediction log's own directory. Not the system temp dir. os.replace is atomic only within a single filesystem; on the Pi the log lives on the SD card and the tempfile must too. tempfile.mkstemp(dir=target_dir) guarantees that.
  • Backfill tool flag scheme. The natural intent — "do the new gate-context patch and leave my partial day alone" — is the bare --from-state. That meant the new flag had to flip OFF the implicit default of recompute-actual. Adding --actual as a second flag restores the combined case without inventing a third "do both" verb. Tradeoff: --actual by itself is harmless redundancy (already the default).
  • File size cap exceeded but not split. services/pv_prediction_logger.py grew from 449 to 615 lines. The 300-line cap is breached, but extracting a PVPredictionLogStore for the IO/upsert/atomic-write surface was out of scope. Worth a follow-up: the IO concerns (read, write, atomic rewrite, schema reconcile) are a natural seam that could become its own class without changing public behavior.

2026-05-25 — PV prediction log uses EG4 todayYielding as primary actual

Files: eg4_client.py, services/pv_prediction_logger.py, app.py, plus two new test files (tests/test_eg4_today_yielding.py, tests/test_pv_prediction_logger_eg4_source.py).

What changed

  • EG4Client gained get_today_yielding_kwh_blocking(timeout=30), which schedules a single get_inverter_energy_async() call on the client event loop, extracts EnergyData.todayYielding (reported by the portal in tenths of kWh), and returns it as kWh. Validation lives in the pure module-level helper _validate_today_yielding().
  • PVPredictionLogger gained an optional get_eg4_client callback. At the post-sunset trigger, it now resolves actual_kwh via the new method _resolve_actual_kwh(): EG4 first, CSV trapezoidal integration only when EG4 returns None. The decision and source are gated-logged under the existing PV_LOG tag.
  • _compute_actual_kwh() now wraps the file iterator in a NUL-stripping generator (_nul_stripped_lines) so a single corrupt line in eg4_battery_log.csv (Pi power loss mid-flush) no longer raises _csv.Error: line contains NUL and abort the entire day.
  • app.py wires get_eg4_client=lambda: battery_service.client — same callback-not-reference pattern WeatherService already uses, for the same reason: BatteryService.refresh_session() replaces the underlying EG4Client.

Non-obvious decisions

  • 0.0 is a real reading, not "missing". The branch in _resolve_actual_kwh is if eg4_value is not None (not truthiness). A fully cloudy day legitimately yields 0 kWh; falling back to CSV in that case would either still return 0 (best case) or, with sensor noise, a small positive value that would corrupt the calibration log. Two dedicated tests guard this — test_resolve_uses_eg4_zero_directly_and_does_not_fall_back and the validator-level test_validate_zero_is_valid_and_returns_zero_kwh.
  • No recovery loop in the blocking call. get_today_yielding_kwh_blocking intentionally does NOT plumb into _attempt_recovery(). It runs once per day at the post-sunset trigger; the background polling thread has already exercised recovery if the session was stale. Adding another recovery path here would race with the polling thread's cooldown timer and could double-spend the 300s re-login budget. CSV fallback handles the rare case where the session is dead exactly at sunset+30.
  • Pure validator separate from the I/O method. _validate_today_yielding() is module-level and side-effect free. The unit tests cover it directly (12 cases) without spinning up the event loop or the EG4 library. The blocking method only adds the loop-scheduling shell — which is covered by one guard-path test plus an injected-fake test at the PVPredictionLogger layer.
  • NUL strip at the line level, not the byte level. The hardened path reads the file as text and only replaces \x00 within each line. Reading bytes-and-decoding would risk mis-decoding partial UTF-8 inside a corrupted region; line-level strip keeps the rest of the day's rows intact while the corrupted line either still parses (when the NULs were sentinel garbage at the end of fields) or gets dropped by parse_battery_row's normal None-on-error contract.
  • File size cap exceeded but not split. Both eg4_client.py (748 lines) and services/pv_prediction_logger.py (448 lines) now exceed the 300-line global cap. Both were already over before this change. The task explicitly scopes core implementation to these two files plus app.py, so splitting was out of scope. Worth a follow-up refactor: eg4_client.py has at least three natural seams (polling/recovery, PV-predict fetch, energy-yield fetch).

2026-05-23 — EG4 PV Energy Predictor + Calibration Tracking

Files: eg4_client.py, utils/eg4_pv_predict.py (new), utils/pv_integration.py (new), services/weather_service.py, services/weather_gate.py, services/pv_prediction_logger.py (new), utils/state_manager.py, config/settings.py, config.yaml, api/weather.py, app.py, static/index.html, static/js/dashboard.js, static/css/dashboard.css, logs/logging.yaml, plus four new test files.

What changed

  • WeatherGate now prefers EG4 portal's own daily PV prediction over the sinusoidal solar_model when the EG4 cache is fresh. The raw prediction is multiplied by a user-tunable safety factor (default eg4_predict_multiplier=0.8) before comparing against the deficit threshold. The solar_model remains as the documented fallback when EG4 is unavailable.
  • WeatherService gained a second cache (EG4 PV-predict) with its own freshness window and a dedicated refresh_eg4_prediction(force=False) method. get_today_forecast() remains cache-only and never blocks on network I/O — refreshes happen on the background poll thread or via POST /api/weather/evaluate_now.
  • EG4Client gained get_latest_pv_predict() (cache read) and refresh_pv_predict_blocking(timeout) (sync entrypoint that schedules an async fetch on the client's event loop and waits). The actual HTTP POST hits /WManage/api/weather/forecast with the X-Requested-With header and form-encoded serialNum; recovery hooks into _attempt_recovery() for HTTP >=400, top-level success: false, and ePvPredict.success != "True".
  • New PVPredictionLogger service runs on its own background thread and appends one row per day to miner_logs/pv_prediction_log.csv about 30 minutes after sunset. It trapezoidally integrates pv_power_w from eg4_battery_log.csv over the local calendar day and pairs the actual with the EG4 raw prediction the gate used that morning, plus the multiplier and decision source.
  • New GET /api/weather/prediction_history?days=N returns the last N rows for the dashboard's Prediction Tracker card.
  • Dashboard: a new eg4_predict_multiplier input on the Parameters card, and a new Prediction Tracker card showing today's EG4 raw / multiplier / expected / source plus the last 7 days as a table with color-coded ratio cells (green 0.7–1.1, yellow 0.5–1.3, red otherwise).

Non-obvious decisions

  • Zero is valid, not missing. parse_pv_predict_kwh(0) returns 0.0, not None. EG4's portal emits zero on confidently-overcast days; conflating that with "missing" would silently flip the safety logic and let the miner run on an empty forecast. The four dedicated tests in test_eg4_pv_predict_parser.py and test_weather_gate_eg4_path.py guard this end-to-end.
  • get_eg4_client callback, not direct reference. BatteryService.refresh_session() replaces the EG4Client instance when the session is recovered. WeatherService takes a lambda: battery_service.client so it always sees the current client; a captured instance would silently go stale exactly when recovery matters most.
  • Precondition split. The gate's "can we evaluate" check used to require both battery_fresh and a fresh Open-Meteo cloud cover. With the EG4 path, only one forecast source has to be fresh — either is enough. Open-Meteo is only required for the fallback. Both unavailable → OUTCOME_FORECAST_STALE (do not advance evaluated_date, retry next tick). This change is what unblocks evaluation on the days when Open-Meteo is down but EG4 is up.
  • Pure helpers split into utils. utils/eg4_pv_predict.py owns the parsing/classification rules; utils/pv_integration.py owns the trapezoidal math + row hygiene. Both are imported by the I/O-heavy services so the math is unit-testable without spinning up the event loop or touching the filesystem. This is the same pattern utils/solar_model.py already follows.
  • Gap policy in the integration. Adjacent samples more than 30 minutes apart discard that pair's contribution (instead of trapezoidally integrating across the gap). Without this rule, a single multi-hour EG4 outage would push the daily total far above the actual harvest. The default threshold is exposed as DEFAULT_GAP_THRESHOLD_SEC so the threshold is auditable; the test suite locks the 30-minute boundary in place.
  • Logger uses gate.evaluated_date == today as a guard. If the gate didn't actually evaluate today (e.g. weather gate disabled, both forecast sources unavailable), the prediction log row writes blanks for the prediction context rather than dragging yesterday's numbers forward. The day's actual_kwh is still recorded — only the prediction side is left blank.
  • Idempotent CSV append. The logger checks both last_pv_log_date in state and the actual last row in the file before appending. Belt + suspenders against the rare case where the state flag is out of sync with the on-disk file (e.g. crash between write and state.save).
  • force=True on evaluate_now triggers a network call. The operator hitting "Force re-evaluate" expects the latest portal reading, not whatever was cached. refresh_eg4_prediction(force=True) serializes via a refresh-lock so back-to-back clicks don't fan out to the EG4 portal.
  • EG4 lib private API. The fetch reuses EG4InverterAPI._get_session() (preferred) or ._session (fallback) to ride the existing JSESSIONID cookie. Both are private surfaces; AttributeError on a future library version is caught and reported as session_lookup_failed rather than crashing the client. The library version pinned in requirements.txt is the contract.
  • eg4_client.py file size. The new fetch + recovery logic grew the file from 401 to 655 lines. Restructuring into a mixin would add indirection without clarity — the new methods are cohesive with the existing I/O surface (login, runtime fetch, battery fetch). Pure parsing is already extracted into utils/eg4_pv_predict.py.
  • Foreman session limitation. The dispatching environment for this session did not expose the Agent tool, so Architect / QA / Platform pre- and post-reviews were performed inline by the Builder against the project's SOLID + design-pattern rules instead of by dedicated subagents. The decision log here doubles as the architect's diff review.

2026-05-23 — Pending-power resolution rewrite (Task 1 of pending-power flywheel)

File touched: static/js/dashboard.js

What changed

  • Frontend pending-power resolution (the "Powering up…" / "Shutting down…" indicator after the user clicks the power toggle) now consumes GET /api/miner/op_status instead of inferring success from MHS 5s, Power Limit, or autocontrol target_w heuristics.
  • Added state.lastOpStatus and a fetch in refreshStatus() that runs before updateMinerStatus() so the resolution branch sees the current op snapshot in the same poll tick.
  • Extracted a pure helper _resolvePendingPowerOutcome(pending, opStatus, nowMs) returning { outcome: 'pending' | 'success' | 'failure' | 'timeout', error? }. No DOM, no fetch — easy to reason about and trivially testable later.
  • Removed state.pendingPowerRetries and both frontend retry-fetch paths (fetch('/api/miner/power_off') / power_on from inside the polling branch). The backend MinerController owns the retry lifecycle; the frontend now only renders outcomes.

Why

The previous heuristic for "powered up" was currentPower >= 0.9 * target_w. On a successful power-on with autocontrol still in its ramp window, target_w can be high but actual Power lags for tens of seconds while upfreq completes — that produced the spurious "Power-up failed — retrying" banner that triggered this bugfix. The backend op_status already publishes the authoritative state (idle / applying / verifying / error) keyed by op_kind and started_at, so the frontend was duplicating logic against weaker signals.

Non-obvious decisions

  • 2-second clock-skew tolerance on the started_at >= pending.start check. Browser Date.now() and the Pi's wall clock can drift slightly; a strict equality would intermittently classify a fresh op as "stale" and keep waiting. 2s is tight enough that a genuinely stale prior op (seconds-to-minutes old) is still rejected.
  • 90-second timeout when no matching op_status entry ever appears. The backend always enqueues an op on power_on / power_off, so the only way to hit this is if the backend itself is wedged or the user clicked the toggle just as the worker thread was draining a previous op. 90s is well past the verify loop's worst case (~30s).
  • Failure / timeout sets the toggle to the OPPOSITE of what the user requested. If "power on" fails, the miner is still off, so the toggle should reflect off. Same for shutdown failure. The previous code passed the requested state, which left the UI lying about the actual miner state.
  • op_status fetched inline inside refreshStatus before updateMinerStatus. Considered parallelising with Promise.all, but kept sequential for now — the extra ~50ms is irrelevant on a 10s poll cadence, and sequential is easier to reason about for a critical-safety path.
  • Frontend retries removed entirely. The MinerController's queue worker has its own verify-and-retry loop (Layer 3). Doubling that up from the frontend caused the original "stop sent twice" race that landed in earlier commits. The frontend now reports what the backend tells it, full stop.

2026-05-23 — User Power Intent (Power toggle bug fix)

Summary

The dashboard Power toggle previously mirrored status.is_off, which made it flip OFF on its own during transient autocontrol verification failures and safety-driven shutdowns. The toggle is a USER COMMAND, not a status indicator. This change introduces a persisted user_power_intent boolean owned by the HTTP layer: the toggle now reflects what the user clicked, not what the miner is doing.

Architecture

  • Ownership: user_power_intent lives in wm_state.json via StateManager.DEFAULT_STATE (default True for forward-compat). MinerService caches the value and exposes a property + a single setter (set_user_power_intent) that persists.
  • The only writers are the two user-facing HTTP endpoints POST /api/miner/power_on and POST /api/miner/power_off in app.py. Service-side code (AutoControl safety stops, MinerController verification) never touches it — structurally enforced because MinerService.power_on() / power_off() only enqueue queue ops; they do not call the intent setter.
  • Defense in depth: POST /api/autocontrol/enable rejects with HTTP 400 + JSON body {"error":"power_off","message":"…"} when intent is False, even though the dashboard already greys out the AC checkbox in this case.
  • Power-OFF cascade: clicking Power-OFF sets intent False AND disables AutoControl if it was enabled (so a tick can't immediately re-power the miner). Clicking Power-ON sets intent True but intentionally does NOT touch AutoControl — the user re-enables AC separately.

Frontend

  • Power toggle checked driven by status.user_power_intent === true, not status.is_off === false.
  • AC toggle disabled when intent is False; CSS class .disabled-by-intent on the label dims it (opacity 0.45) and sets cursor: not-allowed.
  • AC change handler checks resp.ok; on non-2xx it reverts the checkbox to its prior state and surfaces the server-supplied message via addError.

Non-obvious decisions

  • Power-OFF disables AutoControl synchronously in the endpoint (not in set_user_power_intent). The setter only owns the persisted flag; the cascade (disable AC) is HTTP-layer policy because the rule is about user-action semantics, not state-machine invariants. Keeps the setter pure and prevents accidental AC-disables from anywhere else that might want to write intent.
  • Power-ON does NOT auto-enable AutoControl. Per the spec, ON just says "master switch is on"; the user separately decides whether AutoControl should run. Asymmetric to Power-OFF on purpose: turning AC off should not require manually flipping AC, but turning AC on should be an explicit second click.
  • AC toggle's disabled state no longer follows is_off. Previously, the dashboard greyed out AC when the miner was physically off. Now it greys out AC only when user intent is OFF. This means the user can re-enable AC while the miner is physically off but user intent is on (e.g., during a battery-stale safety stop) — which is correct: AC will run its next tick and decide whether to re-issue power_on.
  • Failure-path toggle flicker is acceptable. When a power_on op fails verification, _clearPendingPower(!wasPoweringUp) sets the toggle to OFF for ≤1s, then the next status poll re-drives it from user_power_intent === true. The brief flash reflects the miner's actual transient state and self-heals; no extra logic needed.
  • Forward compatibility via StateManager merge. The Pi's existing wm_state.json doesn't have user_power_intent; StateManager.load() already merges with DEFAULT_STATE, so missing keys get the default True. No migration script. The next save persists the field.
  • state_manager is an optional kwarg on MinerService.__init__ defaulting to None. The production app always passes it, but allowing None lets tests/CLIs instantiate the service without wiring up state. The setter no-ops persistence when state_manager is None (the property still works in-memory).
  • POWER_INTENT log tag enabled (true) during rollout. Set to false in logs/logging.yaml once the feature has been verified in production for a few days.

Tests

tests/test_user_power_intent_state.py — pure StateManager round-trip tests covering default value, persistence, no-clobber across unrelated saves, and forward-compat for state files that predate the field. No service instantiation, no network — runs in milliseconds. Total suite: 87 tests (83 prior + 4 new).
